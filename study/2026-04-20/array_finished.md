## Round 1 테스트

### Q1. 배열이 무엇인지 한두 문장으로 설명하라

답: 여러 데이터들을 순서대로 나열하여 저장하는 자료구조. 

### Q2. 배열에서 `arr[3]`처럼 인덱스로 접근하는 것은 왜 빠른가

답: 주소를 통해 직접 접근하기 때문에 0->1->2->3 으로 가는 것 보다 빠르다

### Q3. 배열에서 특정 값을 찾는 탐색은 보통 왜 `O(n)`인가

답: 배열안의 값이 n개일 때 처음부터 탐색 시 최악의 경우 n번 탐색하기 때문이다. 찾는 값의 개수가 주어지지 않은 경우에는 확실하게 n이다.

### Q4. 배열 중간에 값을 삽입할 때 왜 느려질 수 있는가

답: LinkedList처럼 중간에 값을 삽입할 수 있는 구조가 아니기 떄문에 새로운 배열을 처음부터 만들기 떄문이다.

### Q5. 배열과 연결 리스트의 가장 큰 차이 한 가지를 설명하라

답: 배열은 조회가 빠르고 연결리스트는 노드들의 조합으로 이루어져 조회가 느리다.

## Round 1 피드백

- Q1: "순서대로 나열"은 맞지만, 배열의 핵심은 `연속된 메모리 공간`과 `인덱스 접근`이다. 이 두 요소가 들어가면 더 정확하다.
- Q2: 방향은 맞다. 더 정확히는 `시작 주소 + 인덱스 * 원소 크기`로 바로 위치를 계산할 수 있어서 빠르다.
- Q3: 잘 설명했다. `찾는 값의 개수`보다 `원하는 값이 어디 있는지 모르기 때문에 처음부터 비교해야 한다`가 핵심이다.
- Q4: 이 답이 가장 아쉽다. 배열 중간 삽입은 보통 "새 배열을 처음부터 만드는 것"이 본질이 아니라, `삽입 위치 뒤의 원소들을 한 칸씩 뒤로 밀어야 해서` 느리다. 동적 배열 구현에서는 공간이 꽉 찼을 때만 새 배열 복사가 추가로 발생할 수 있다.
- Q5: 맞는 말이다. 다만 더 좋은 비교는 `배열은 인덱스 접근이 빠르고, 연결 리스트는 중간 삽입/삭제가 유리할 수 있다`처럼 양쪽의 강점을 함께 적는 것이다.

## 보강 정리

### 한 줄 정의

배열은 같은 종류의 데이터를 연속된 메모리 공간에 저장하고, 인덱스로 빠르게 접근하는 자료구조다.

### 핵심 포인트

- 인덱스 접근은 빠르다
- 탐색은 보통 순차적으로 봐야 해서 느릴 수 있다
- 중간 삽입/삭제는 뒤 원소들을 밀거나 당겨야 해서 비용이 든다
- 연결 리스트는 중간 삽입/삭제 구조가 다르다

### 연산 복잡도

- 접근: `O(1)`
- 탐색: `O(n)`
- 맨 끝 삽입: 상황에 따라 `O(1)` 또는 동적 배열이면 평균 `O(1)`
- 중간 삽입: `O(n)`
- 중간 삭제: `O(n)`

### 연결 리스트와 비교

- 배열: 인덱스 접근이 빠르다
- 배열: 중간 삽입/삭제가 비싸다
- 연결 리스트: 인덱스 접근이 느리다
- 연결 리스트: 위치를 이미 알고 있다면 삽입/삭제가 유리할 수 있다

## Round 2 재테스트

### Q1. 배열의 가장 큰 장점 1개와 단점 1개를 적어라

답: 장점은 인덱스 접근 속도가 O(1)이라 빠르고 단점은 중간 삽입이 느리다. (O(n))

### Q2. 배열의 접근이 `O(1)`인 이유를 메모리 관점에서 짧게 적어라

답: 배열의 주소가 a라고 할 때 n번째 값을 찾는다면 메모리의 a+(배열 한칸의 크기)*(n-1) 위치로 직접 접근하면 되기 때문이다.

### Q3. 배열의 중간 삽입이 `O(n)`인 이유를 적어라

답: 중간에 값을 넣으면 그 뒤의 값들을 모두 한칸 씩 밀어내야 하기 때문에. 최악의 경우를 기준으로 O(n)

### Q4. 연결 리스트보다 배열이 더 적합한 상황을 1개 적어라

답: 넣을 때 계속 뒤에 넣은 다음에 삭제/수정 없이 조회만 하는 경우. 로그같은걸 찍을 때 배열로 두면 시간을 기준으로 빠르게 접근이 가능하다.

## Round 2 피드백

- Q1: 좋다. 장점과 단점을 복잡도까지 포함해서 잘 연결했다.
- Q2: 충분히 정확하다. 실제 구현에서는 보통 `시작 주소 + 인덱스 * 원소 크기` 형태로 이해하면 된다.
- Q3: 정확하다. 핵심 이유를 잘 짚었다.
- Q4: 적절한 예시다. 특히 `조회가 많고 인덱스 접근이 중요한 경우`라는 기준을 잡은 점이 좋다.
