## Round 1 피드백

- Q1: 방향은 괜찮지만 `동기 = 스레드를 점유한 상태`로만 잡으면 좁다. 동기의 핵심은 `요청한 쪽이 결과를 받을 때까지 흐름을 이어서 기다린다`는 점이다.
- Q2: 비동기를 `이벤트와 콜백을 이용하는 작업`으로만 잡으면 구현 방식에 묶인다. 핵심은 `작업 완료를 나중에 통지받는다`이다. 콜백, 이벤트, Promise는 그 방법들이다.
- Q3: 여기서 개념이 좀 섞였다. `동기면 스레드가 작업 수만큼 필요하다`는 건 일반화하기 어렵다. 또 `비동기가 CPU 비용이 더 필요하다`도 항상 맞지는 않는다. 핵심 차이는 `기다리는 방식`과 `완료 통지 방식`이다.
- Q4: 좋다. 타이머 예시로 비동기 감각을 잘 잡았다.
- Q5: 방향은 좋다. 다만 `I/O 대기시간을 없앤다`보다는 `I/O 대기 동안 호출한 쪽이 막히지 않고 다른 일을 할 수 있게 한다`가 더 정확하다. 대기 자체가 사라지는 것은 아니다.

## 보강 정리

### 한 줄 정의

- 동기: 요청한 작업이 끝날 때까지 기다린 뒤 다음 작업을 진행하는 방식
- 비동기: 작업을 맡겨두고, 끝나기 전에 다른 일을 하다가 완료 통지를 받는 방식

### 핵심 포인트

- 동기의 핵심은 `순서대로 기다림`이다
- 비동기의 핵심은 `끝날 때까지 막히지 않음`이다
- 비동기는 결과를 나중에 콜백, 이벤트, Promise, await 같은 방식으로 받는다
- 비동기는 동시성, 병렬성과 관련될 수 있지만 같은 뜻은 아니다

### 예시

- 동기: 카페에서 주문하고 음료가 나올 때까지 카운터 앞에서 계속 기다린다
- 비동기: 주문만 하고 진동벨을 받은 뒤 자리에 가서 다른 일을 하다가 알림이 오면 음료를 받는다

### 헷갈리기 쉬운 비교

- 동기/비동기: 결과를 기다리는 방식 차이
- 동시성/병렬성: 여러 작업을 처리하는 방식 차이

- 동기라고 해서 항상 느린 것은 아니다
- 비동기라고 해서 항상 동시에 여러 CPU 작업이 도는 것은 아니다

## 심화 정리

### 1. 가장 먼저 구분할 것

- 동기/비동기: `결과를 언제 받는가`
- 블로킹/논블로킹: `호출한 스레드가 멈추는가`

이 둘은 관련은 있지만 같은 말이 아니다.

예를 들어:

- 동기 + 블로킹: 요청 후 끝날 때까지 그대로 멈춘다
- 동기 + 논블로킹: 바로 돌아오지만, 결과가 준비될 때까지 계속 확인해야 한다
- 비동기 + 논블로킹: 맡겨두고 나중에 완료 통지를 받는다

### 2. 비동기 I/O가 왜 필요한가

CPU는 계산은 빠르지만, 디스크 읽기나 네트워크 응답은 상대적으로 느리다.

동기 블로킹 방식이면:

1. 프로그램이 파일 읽기나 소켓 읽기를 요청한다
2. 커널에 시스템 콜을 보낸다
3. 데이터가 올 때까지 호출한 스레드가 기다린다
4. 그동안 그 스레드는 다른 일을 못 한다

비동기 방식이면:

1. 프로그램이 읽기 작업을 등록한다
2. 커널 또는 런타임이 이 작업을 관리한다
3. 호출한 쪽은 바로 다음 일로 넘어간다
4. 작업이 끝나면 완료 이벤트, 콜백, Future, Promise 같은 형태로 결과를 전달받는다

핵심은 `대기 시간 자체를 없애는 것`이 아니라 `대기 때문에 호출한 쪽이 멈추지 않게 하는 것`이다.

### 3. 커널은 여기서 무슨 일을 하나

비동기 I/O에서 커널은 보통 아래 역할을 맡는다.

- 디스크, 네트워크 장치 같은 실제 I/O 장치와 통신한다
- 어떤 파일 디스크립터나 소켓이 읽기/쓰기 가능한지 추적한다
- 준비된 이벤트를 프로그램에 알려준다
- 완료된 작업 결과를 버퍼에 채워두거나 완료 신호를 준다

리눅스 기준으로 생각하면:

- `select`, `poll`, `epoll`은 "어떤 소켓이 준비됐는지 알려주는 방식"에 가깝다
- `io_uring`은 더 직접적인 비동기 I/O 모델에 가깝다

즉, 예전 방식은 `준비됐는지 물어보는 모델`이 강했고, 더 발전한 방식은 `작업을 제출하고 완료를 받는 모델`에 가까워진다.

### 4. 이벤트 루프는 무슨 일을 하나

이벤트 루프는 "나중에 처리할 일들"을 계속 돌면서 처리하는 관리자라고 보면 된다.

보통 흐름은 이렇다.

1. 처리할 콜백/태스크 큐를 본다
2. 준비된 I/O 이벤트가 있는지 확인한다
3. 완료된 작업의 콜백을 실행한다
4. 새로 생긴 작업을 다시 큐에 넣는다
5. 이 과정을 계속 반복한다

Node.js를 예로 들면:

- 자바스크립트 코드는 보통 한 스레드에서 돈다
- 네트워크 I/O나 타이머 완료는 이벤트 루프가 받아서 콜백 큐에 넣는다
- 메인 흐름은 "끝난 작업의 후속 처리"만 실행한다

그래서 `한 스레드인데도 여러 연결을 처리하는 것처럼 보이는` 것이다.

### 5. 비동기가 CPU 작업을 빠르게 해주는가

보통은 아니다.

- 비동기는 `I/O 대기`에 강하다
- 무거운 CPU 계산은 스레드나 프로세스를 따로 써야 한다

예를 들어:

- 파일 다운로드 1만 개 처리: 비동기 모델이 유리할 수 있다
- 이미지 압축, 머신러닝 계산, 암호화 연산: CPU 작업이라 비동기만으로 빨라지지 않는다

### 6. 네가 말로 설명할 때 쓸 수 있는 틀

`동기/비동기는 여러 작업을 동시에 하느냐가 아니라, 결과를 기다리는 방식의 차이다.`

`동기는 호출한 쪽이 작업이 끝날 때까지 흐름을 붙잡고 있고, 비동기는 작업을 맡긴 뒤 나중에 완료 통지를 받아 이어서 처리한다.`

`특히 비동기는 네트워크나 디스크처럼 기다리는 시간이 긴 I/O 작업에서 유리하고, 이때 커널과 이벤트 루프가 완료된 작업을 알려주는 역할을 한다.`

## 용어 연결 지도

### 1. 먼저 층위를 나눠서 봐야 한다

- 동기/비동기: `결과를 언제 받는가`에 대한 개념
- 블로킹/논블로킹: `호출한 스레드가 멈추는가`에 대한 개념
- Future/Promise: `나중에 올 결과를 코드에서 어떻게 표현하는가`
- select/poll/epoll: `리눅스 커널에게 어떤 I/O가 준비됐는지 어떻게 묻는가`
- 이벤트 루프: `준비된 작업과 콜백을 계속 처리하는 실행 구조`
- 커널: `실제 I/O 장치와 통신하고 준비 상태를 관리하는 운영체제 핵심 부분`

### 2. 리눅스에서 네트워크 I/O가 일어날 때 큰 흐름

1. 프로그램이 소켓에 대해 읽기/쓰기를 요청한다
2. 이 요청은 시스템 콜을 통해 커널로 들어간다
3. 커널은 NIC, 디스크, 메모리 버퍼 같은 실제 자원과 상호작용한다
4. 아직 데이터가 준비되지 않았으면, 지금 당장 읽을 수 없다는 상태를 관리한다
5. 데이터가 도착하면 커널은 "이 소켓은 이제 읽을 준비가 됐다"는 상태를 표시한다
6. 사용자 프로그램은 이 준비 상태를 보고 후속 콜백이나 로직을 실행한다

즉, `실제 준비 상태를 아는 쪽은 커널`이고, `그 준비된 일을 처리하는 쪽은 사용자 공간의 이벤트 루프`다.

### 3. select, poll, epoll은 각각 뭔가

셋 다 "어떤 파일 디스크립터가 지금 읽기/쓰기 가능한가"를 알아내기 위한 리눅스 계열 방식이다.

- `select`: 오래된 방식. 감시할 대상을 매번 넘기고, 개수 제한과 복사 비용이 있다.
- `poll`: `select`보다 조금 일반화됐지만, 여전히 많은 대상을 매번 훑는 비용이 있다.
- `epoll`: 감시 대상을 커널 쪽에 등록해두고, 준비된 것 위주로 효율적으로 받는다.

핵심 차이:

- `select/poll`: "매번 전체를 검사하는 느낌"
- `epoll`: "관심 있는 대상을 등록해두고 준비된 것만 받는 느낌"

그래서 연결 수가 많아질수록 보통 `epoll`이 유리하다.

### 4. 이벤트 루프의 위치

이벤트 루프는 보통 `사용자 공간 런타임` 안에 있다.

예:

- Node.js의 libuv
- Python의 asyncio 이벤트 루프
- 브라우저의 JS 이벤트 루프

이벤트 루프가 직접 네트워크 카드를 다루는 것은 아니다.
그건 커널이 한다.

이벤트 루프는 보통 이렇게 일한다.

1. 지금 실행할 태스크가 있으면 실행한다
2. 없으면 커널에게 "준비된 I/O 있나?"라고 묻거나 기다린다
3. 준비된 소켓/타이머/이벤트를 받는다
4. 그에 연결된 콜백이나 후속 작업을 실행 큐에 넣는다
5. 다시 반복한다

즉:

- 커널: 준비 여부를 관리
- 이벤트 루프: 준비된 후속 작업을 실행

### 5. Future와 Promise는 왜 필요한가

비동기에서는 결과가 지금 없고 `나중에` 온다.
그 "나중에 올 값"을 코드에서 다룰 방법이 필요하다.

- `Future`: 미래에 완료될 작업의 결과를 나타내는 객체
- `Promise`: 미래에 성공 또는 실패로 끝날 값을 약속하는 객체

실무에서는 둘을 거의 비슷한 역할로 이해해도 초반에는 큰 문제 없다.

핵심은 이거다.

- 지금 값이 없음
- 하지만 나중에는 값이 생길 예정
- 그 값을 받을 후속 처리도 같이 연결할 수 있음

그래서 콜백 지옥 대신:

- `then`
- `catch`
- `await`

같은 문법으로 비동기 흐름을 더 읽기 좋게 만든다.

### 6. Promise/Future와 이벤트 루프 관계

이 둘은 같은 것이 아니다.

- 이벤트 루프: 실행을 돌리는 관리자
- Promise/Future: 나중에 올 결과를 담는 표현

비유하면:

- 이벤트 루프 = 우체국 분류 시스템
- Promise/Future = 배송 예정 송장

배송 예정 송장만 있다고 택배가 움직이는 것은 아니고,
분류 시스템만 있다고 각 물건의 약속 정보가 생기는 것도 아니다.

### 7. 커널이 "이벤트를 발생시킨다"는 말의 뜻

이 표현은 완전히 틀린 말은 아니지만, 초반에는 이렇게 이해하는 편이 안전하다.

- 커널은 I/O 대상의 준비 상태를 관리한다
- 프로그램은 `epoll_wait` 같은 호출로 준비된 항목을 받는다
- 사용자 공간 런타임은 그걸 이벤트처럼 해석해서 콜백을 실행한다

즉, "이벤트"라는 말은 커널 내부 신호 자체라기보다 `준비된 상태를 사용자 공간이 이벤트처럼 소비하는 모델`에 가깝다.

### 8. 네가 지금 단계에서 잡아야 할 핵심 문장

`리눅스에서 비동기 I/O는 보통 커널이 소켓이나 파일의 준비 상태를 관리하고, 사용자 공간의 이벤트 루프가 준비된 작업의 후속 콜백을 실행하는 구조로 이해하면 된다.`

`select, poll, epoll은 준비된 I/O를 알아내는 방법이고, Future/Promise는 나중에 올 결과를 코드에서 표현하는 방법이다.`

- 일상 예시나 프로그래밍 예시로 구분할 수 있다
- 동시성/병렬성과 다른 개념임을 설명할 수 있다
- 블로킹/논블로킹과 구분할 수 있다
- 커널과 이벤트 루프의 역할을 대략 설명할 수 있다

## 최종 정리용 빈칸

- 아직 헷갈리는 점: `동기/비동기`, `블로킹/논블로킹`, `이벤트 루프`, `Promise/Future`를 처음에는 한 덩어리로 섞어서 봤다
- 다시 확인할 개념: 커널은 준비 상태를 관리하고, 이벤트 루프는 준비된 FD를 받아 후속 작업으로 연결한다
- 통과 여부: 통과

## 최종 정리

- 동기와 비동기의 핵심 차이는 여러 작업을 동시에 하느냐가 아니라 결과를 기다리는 방식에 있다
- 리눅스 비동기 I/O에서는 커널이 소켓 FD의 준비 상태를 관리하고, 이벤트 루프가 `epoll_wait`로 준비된 FD를 받아 후속 읽기 처리와 콜백 실행으로 이어준다
- Promise/Future는 커널 기능이 아니라 사용자 공간에서 미래 결과와 완료 상태를 표현하는 객체다
