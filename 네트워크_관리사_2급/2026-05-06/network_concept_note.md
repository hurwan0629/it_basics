# 이것저것 알아보기
## IRC
Internet Reply Chat의 약자로 인터넷에서 여러 사람이 실시간으로 채팅하기 위한 오래된 프로토콜이다.
사용자가 [IRC 서버]에 접속해서 채널에 들어간 뒤 메시지를 주고 받게 하는 방식이였다.

`IRC Client`와 `IRC Server` 이 있으며 `Channel`, `Nickname`, `Operator`, `Private Message` 등의 갠ㅁ이 존재하였다.

흐름으로는
1. IRC 클라이언트로 서버에 접속 (url)
2. 닉네임을 정하고 채널에 들어가기
3. 메시지를 보내기

특징으로는
### 장점
- 구조가 단순
- 텍스트 기반이라 가벼움
- 오래된 오픈소스 커뮤니티에서 아직도 사용중
- 봇을 만들기 쉬움
### 단점
- UI가 불친절하다
- 파일 기능에 약하다.
- 인증/보안/저장 기능이 현대 서비스보다 약하다
- CLI기반 인터페이스이다.

## HEVC / H.265
HEVC는 `High Efficiency Video Coding`의 약자이며 `H.256`으로도 불린다.
쉽게 말하면 동영상 용량을 줄이기 위한 압축 기술이다.

영상은 원본 그대로 저장하면 용량이 매우 크기 때문에 보통 영상을 압축해서 저장 또는 전송한다.
이때 사용하는 압축 규칙이 **코덱**이며 HEVC/H.256은 그중 하나이다.

H.264는 AVC(Advenced Video Coding)라고 부르는데 H.256은 H.265보다 더 적은 비트레이트(전송 비트량)를 사용하기 때문에 4K, HDR, 고화질 스트리밍에서 자주 쓰이게 됩니다.

반대로 AVC는 연산 속도가 상대적으로 빠르기 때문에 일반 스트리밍, FHD에서 사용됩니다. (AVC가 호환성이 더 좋습니다.)

### HEVC(H.256)의 주요 압축 아이디어
- 한 프레임 안에서도 비슷한 색상과 패턴이 반복되면 이를 줄인다.
- 프레임끼리 비슷한 부분을 재사용하며 차이점을 중심으로 저장한다.
- 움직임을 기록한다. 특정 블록이 n방향으로 m만큼 이동했다 와 같이 저장한다.

이때 I-frame를 기준이 되는 완전한 프레임
P-frame를 이전 프레임과의 차이점을 저장
B-frame가 앞뒤 프레임을 참고해서 저장
하는 프레임들입니다.

## SIP
Session Initial Protocol로 **음성통화**, **영상통화**같은 통신 세션을 시작하고 관리하기 위한 프로토콜입니다.
대표적으로 인터넷 전화, VoIP(Voice over Internet Protocol), 화상회의에 사용됩니다.

SIP는 직접 음성 데이터를 보내는 것이 아닌, 통화 연결을 설정하고 종료하는 신호 프로토콜입니다.

실제 음성/영상 데이터는 보통 RTP같은 다른 프로토콜이 담당합니다.

SIP의 메시지로는
- INVITE (통화요청)
- 180 Ringing - 상대방 벨 울리는중
- 200 OK - 요청 수락
- ACK - 수락 확인
- BYE - 통화 종료
- REGISTER - 사용자의 현재 위치 등록

SIP의 구성 요소로는
- User Agent Client(UAC): 요청을 보내는 쪽
- User Agent Server(UAS): 요청을 받는 쪽
- Proxy Server: 요청 중개
- Registrar Server: 사용자의 현재 위치 등록
- Location Server: 사용자가 어디에 있는지 저장
- SIP URI: SIP주소

SIP 주소는 이메일과 비슷하게 생겼습니다.
- 통화 세션 관리: SIP
- 음성/영상 데이터 전달: RTP
- 미디어 정보 협상: SDP
- 암호화된 미디어: SRTP
로 이루어집니다.

### SIP의 장점
- SIP는 인터넷 기발 전화 시스템에 적합합니다.
- 텍스트 기반이라 구조를 읽기 쉽다.
- 확장성이 좋다
- VoIP, 콜센터, 화상회의 등에서 활용된다.
### SIP의 단점
- NAT 환경에서 복잡해질 수 있다.
- 실제 미디어 전송은 다른 프로토콜과 함께 이해해야한다.
- 보안 설정이 필요하다
- SIP 서버, 프록시, 등록 서버 등 구조가 복잡해질 수 있다.

# 터널링
## VPN의 기초
VPN을 이해할 때에는 3가지로 분리 해야한다.
- 터널링: 원래 패킷을 다른 패킷 안에 감싸서 보내는 방식 (L2TP, GRE)
- 암호화/인증: 중간에서 못보게 암호화하고 상대를 확인하는 방식 (IPsec, SSL/TLS)
- VPN 프로토콜: 터널링 + 인증 + 암호화를 조합한 VPN 방식 (IPsec VPN, SSL VPN, L2TP/IPsec)

## 터널링
터널링은 어떤 네트워크 패킷을 다른 패킷 안에 넣어서 보내는 기술이다.
원래 보내려던 데이터 바깥에 새로운 주소지를 만들어 보내는 방식이다.
보통 내 컴퓨터 공인 IP는 특정 주소에 갈 수 없게 막히기 떄문에 이것을 회사 등의 VPN 서버 공인 IP로 보낸 뒤 VPN내부 IP에서 회사의 서버로 보내는 방식과 같다.
터널링의 핵심 효과로는
- 사설망 접근
- 주소 은닉
- 캡슐화
- 암호화 (터널 안 데이터를 암호화하면 VPN이 된다.)

하지만 터널링 자체는 "감싸서 보내는 기술"이고 암호화는 내용을 못보게 만드는 기술이다.
L2TP는 터널링은 하지만 자체 암호화는 약하거나 없다.
그래서 IPsec과 같이 사용하게 된다.

## IPsec
IPsec는 IP 계층에서 패킷을 암호화하고 인증하는 보안 프로토콜 묶음이다.
IPsec는 단일 프로토콜 보다는 여러 보안 기술의 집합에 가깝다.

IPsec는 주로 3계층에 존재한다.
이를 통해서 IPsec는 특정 어플리케이션 뿐 아니라 IP패킷 자체를 보호할 수 있다.

IPsec의 주요 목적은 3가지이다.
- 암호화: 패킷 내용을 중간에서 못보게함
- 무결성 검증: 중간에 데이터가 변조됐는지 확인
- 인증: 통신 상대가 진짜인지 확인
이를 통해서 IPsec VPN에서는 내 컴퓨터와 VPN서버가 서로 신뢰할 수 있는 상대인지 확인하고 이후 패킷을 암호화해서 보낸다.

### IPsec의 주요 2가지 모드
- Transport Mode: IP패킷의 내용 부분만 보호 - 호스트 간 보안에 쓰임
- Tunnel Mode: IP 패킷 전체를 새 IP패킷 안에 넣어서 보호 - VPN에 쓰임
VPN에서 중요한건 대부분 Tunnel Mode이다.

Transport Mode는 기존 IP헤더는 대체로 유지되고 그 안의 데이터부분을 보호한다.
IPsec 적용 후에는 겉 IP주소는 그대로 보이고 내부 데이터를 보호한다.

Tunnel Mode는 원래 IP 패킷 전체를 암호화해서 새 IP 패킷 안에 넣는다.
이를 통해서 외부에서 볼때에는 사용자 공인 IP에서 VPN서버 공인 IP로 가는것으로 보이지만 터널 안쪽에는 사용자 VPN 내부 IP -> 회사 내부 서버 IP로 가는 것처럼 보이게된다.

### IPsec의 AH와 ESP
IPsec에는 대표적으로 두가지 보안 프로토콜이 존재한다.
- AH: 인증/무결성
- ESP: 암호화/인증/무결성

AH는 Authentication Header로 주로 인증과 무결성을 제공하지만 암호화는 하지 않는다.
이 때문에 데이터의 조작은 탐지 가능하지만 내용 자체를 숨기지는 못한다. 이때문에 ESP가 더 많이 쓰인다.

ESP는 Encapsulating Security Payload로 주로 암호화, 인증, 무결성을 제공한다.
VPN에서 일반적으로 더 중요한건 ESP이다. VNP에서의 일반적인 ESP 패킷은 다음과 같이 생겼다
`[새 IP헤더 [ESP 헤더 [암호화된 원래 패킷] [ESP 인증 정보]]]`

### IPsec 연결 과정: IKE
IPsec는 암호화 자체만으로 끝나지 않는다.
**통신 전에 양쪽의 합의를 한다.**
- 어떤 암호화 알고리즘을 쓸 것인지
- 키는 어떻게 만들 것인지
- 인증 방식
- 연결 유지 기간
등을 합의하며 이 협상을 담당하는 것이 IKE(Internet Key Exchange)이다.
요즘에는 IKEv2가 많이 쓰인다.

IKE는 다음과 같은 작업을 한다.
- 상대 인증: 비밀번호, 사전 공유 키, 인증서 등으로 확인
- 암호화 방식 협상: AES, SHA 등 알고리즘 선택
- 키 교환: 실제 암호화에 쓸 키 생성
- SA 생성: 보안 연결 상태 생성
여기에서 SA는 Security Association이다. (보안 호환)

## PPTP
PPTP는 Point-to-Point  Tunneling Protocol의 약자이다.
오래된 VPN 터널링 프로토콜로 Microsoft 환경에서 과거에 많이 사용되었다.
PPTP의 구조는 보통 다음 조합으로 동작한다.
`PPP + GRE + 제어 연결`
- PPP: 사용자 인증과 링크 구성
- GRE: 데이터를 터널링
- TCP 1723: PPTP 제어 연결

### PPTP의 장점
- 설정이 간단하다
- 속도가 빠른 편이다
- 오래된 OS에서도 지원된다
- 구조가 비교적 단순하다
### PPTP의 단점
- 보안이 약해서 일반적으로 새 VPN 구축에는 권장되지 않는다. 특히 과거 PPTP에서 많이 쓰인 MS-CHAPv2 인증방식이 취약하다는 평가를 받았다.

이때문에 PPTP라는 터널링 프로토콜은 알아만 두고 잘 사용하는 일은 없을 것이다.

## L2TP란
L2TP는 Layer 2 Tunneling Protocol의 약자이다.
이름 그대로 2계층 프레임을 터널링하기 위한 프로토콜이다.

L2TP 자체는 강한 암호화를 제공하진 않기 떄문에 `L2TP + IPsec` 조합을 사용한다.

방식은
`원래 데이터 -> L2TP 터널링 -> IPSEC 암호화` 방식으로 이루어진다.

L2TP는 보통 UDP 1701을 이용한다.
- L2TP: UDP 1701
- IKE: UDP 500
- NAT-T: UDP 4500
- ESP: IP protocol 50
NAT 환경에서는 ESP가 직접 통과하기 어려울 수 있어서 UDP 4500을 쓰는 NAT-T가 더 중요하다.
